base_local_planner

发表于 2024-06-19 分类于 路径规划

ROS文档解读
ref: https://wiki.ros.org/base_local_planner

Overview

这个包给movebase提供控制器。用来计算满足动力学的轨迹。

map grid

每次进行控制，都会更新局部地图，且将全局规矩映射到局部地图上。这个地图用来存储轨迹。

oscillation suppression：

三个轴上的值在正负之间chosen consecutively，就发生了震荡。当机器人朝一个方向移动时，下一个循环相反方向则被禁止，直到移动足够的距离才重置。

dwa_local_planner

1 TrajectorySampleGenerator

接口类，定义函数nextTrajecotry.

2 TrajecotryCostFunction

接口类，定义函数scoreTrajectory，输入轨迹，输出分数，负值意味着轨迹不可用。

3 SimpleScoredSamplingPlanner

一个简单的轨迹搜索实现。维护一个TrajectorySampleGenerator和一个TrajectoryCostFunction列表。

Helper classes

1 LocalPlannerUtil

帮助类提供一个movebase上下文环境下的局部规划器通用函数。管理当前全局目标，当前运动限制，当前局部代价地图。

2 OdometryHelperRos

提供odometry信息

3 LatchedStopRotateController

close enough to the goal. 理想的局部规划器可以使机器人精确的停下来，但是由于传感器噪声和执行器不确定性，可能会超过目标点。这个类就是让尽可能靠近目标。主要函数：computeVelocityCommandsStopRotate，

Cost function

基于连续的轨迹进行计算代价值。连续轨迹是我自己给的称呼，例如Astar算法得到的结果就是连续的，经过裁剪（只保留直线两端的顶点）的轨迹则不是连续稠密的，这种轨迹不能直接用于计算代价值。

1 ObstacleCostFunction

根据可观察到的障碍物计算轨迹的代价值。如果返回负值，则这个轨迹穿过了障碍物。需要使用world_model

2 MapGridCostFunction

这个类评估轨迹和全局路径的接近程度，或者和目标点的接近程度。尝试使用相同的预计算距离地图来优化路径。在dwa_local_planner中，这个类被多次实例化：

  • 保证轨迹靠近全局路径
  • 保证机器人朝向局部目标前进
  • 保证机器人前向朝着局部目标

3 OscillationCostFunction

帮助减少确定性的震荡，如果上一次切换累计了一定距离，则返回运动方向的负代价值。

auto/decltype

发表于 2024-06-15 分类于 CPP
  1. “自动”就是让计算机去做，而不是人去做，相对的是“手动”
  2. “类型”指的是操作目标，出来的是编译阶段的类型，而不是数值
  3. “推导”就是演算、运算，把隐含的值给算出来

auto 让代码干净整齐了很多，不用去写那些复杂的模板参数了。但如果你把“自动类型推导”理解为仅仅是简化代码、少打几个字，那就实在是浪费了 C++ 标准委员会的一番苦心。auto 还避免了对类型的“硬编码”

使用auto需要注意的地方：

  • auto 的“自动推导”能力只能用在“初始化”的场合
  • 如果不是初始化的形式，只是“纯”变量声明，那就无法使用 auto
  • auto 总是推导出“值类型”，绝不会是“引用”
  • auto 可以附加上 const、volatile、*、& 这样的类型修饰符，得到新的类型。

decltype

auto 只能用于“初始化”，而这种“向编译器索取类型”的能力非常有价值，仅仅用来初始化有点屈才了。因此还有另外一个自动类型推导的关键字：decltype。

decltype 的形式很像函数，后面的圆括号里就是可用于计算类型的表达式（和 sizeof 有点类似）。

decltype 不仅能够推导出值类型，还能够推导出引用类型，也就是表达式的“原始类型”。所以，你完全可以把 decltype 看成是一个真正的类型名，用在变量声明、函数参数 / 返回值、模板参数等任何类型能出现的地方，只不过这个类型是在编译阶段通过表达式“计算”得到的。

缺点：实际上，它也有个缺点，就是写起来略麻烦，特别在用于初始化的时候，表达式要重复两次（左边的类型计算，右边的初始化），把简化代码的优势完全给抵消了。
所以在C++14中增加了decltype(auto).

最佳实践

auto的最佳实践，range-based-for，适合在容器遍历时使用，不需要关心容器的元素类型、迭代器返回值和首末位置，为了运行效率，一般使用引用类型：const auto&或者auto&。

它是auto的高级形式，更侧重于编译阶段的类型计算，常用于泛型编程。

ROS功能包之间引用

发表于 2024-06-02 更新于 2024-07-21 分类于 ROS

在ROS中，项目的组织架构以功能包的形式为主。在常见的开发过程中，一般会有各个包都会使用的函数或者消息类型的定义，这个时候就需要定义一个公共的功能包来作为库使用。而这个库怎么然其他功能包使用便是较为关键的一步。下面将逐步演示如何进行创建

1. 创建包

创建common，a两个包，package_common包中定义一些其他包中会使用到的公共函数，a和b是两个正常的功能包

在本文中，不再使用ROS常见的文件夹组织结构（src, include等），而是按照cpp文件的具体功能放置于具体的文件夹中，例如common、utils、math等。

关键是CMakeLists.txt的写法，这里我按照CMakeLists一步一步解释：

  1. 项目相关定义，这里没什么好说的，按照ROS正常流程来
  2. catkin_package这个宏是用来用于定义包的导出信息，使其他包可以找到并使用它。
  • INCLUDE_DIRS: 这个包导出的目录，别的包在引用时，就会包含这个目录。例如a包要引用AddFunction.h文件，就会在这里路径中查找。我喜欢使用${CMAKE_SOURCE_DIR}来给出工作空间路径，而不是相对功能包的路径。
  • LIBRARIES: 用于指定要导出的库。其他依赖该包的包可以链接这些库，本例中导出package_common_math这个库， 当然这里也可以导出多个库。
  • CATKIN_DEPENDS: 用于指定该包依赖的其他 catkin 包。因为目前该包没有引用ros.h，所以这里暂且不使用
  • DEPENDS: 用于指定该包依赖的非 catkin 包或系统库。一般不管
  1. 本包编译配置

  • include_directories 头文件路径设置。在cpp文件中，引用h文件的路径我给的是相对工作空间的路径，因此加上了${CMAKE_SOURCE_DIR}表示工作空间路径

    #include “package_common/math_lib/AddFunction.h”

  • add_library 编译库，STATIC生成静态文件，如果使用PUBLIC则生成动态库文件。cpp文件的路径相对于当前功能包

  • target_link_libraries 链接库， 将编译出来的库文件和ROS链接
